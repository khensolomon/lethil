# r2.py

Cloudflare R2 backup/restore tool for Docker Swarm apps.

A single Python script for backing up MySQL databases to R2, restoring them on app init, and copying arbitrary files between local folders and R2 buckets.

- **Source:** https://github.com/khensolomon/lets/blob/make/server/r2.py
- **Dependencies:** Python 3 + `boto3`

---

## Install

### 1. Install boto3

The script needs `boto3`. Pick the line for your system:

| System          | Command                                          |
| --------------- | ------------------------------------------------ |
| Debian / Ubuntu | `sudo apt install python3-boto3`                 |
| Other           | `sudo pip install boto3 --break-system-packages` |

That's it — no venv, no activation.

### 2. Download the script

```bash
sudo wget https://raw.githubusercontent.com/khensolomon/lets/make/server/r2.py -O /usr/local/bin/r2.py
sudo chmod +x /usr/local/bin/r2.py
```

### 3. Create the config

```bash
sudo mkdir -p /opt/bucket/storage/access
sudo nano /opt/bucket/storage/access/r2.conf
sudo chmod 600 /opt/bucket/storage/access/r2.conf
```

Minimum contents (replace with real values):

```ini
R2_ACCOUNT_ID=your_cloudflare_account_id
R2_ACCESS_KEY_ID=your_r2_access_key_id
R2_SECRET_ACCESS_KEY=your_r2_secret_access_key
R2_ENDPOINT=https://your_cloudflare_account_id.r2.cloudflarestorage.com
R2_BUCKET=storage
```

Optional tuning:

```ini
RETENTION_KEEP=7        # how many timestamped MySQL dumps to keep on prune
COMPRESSION_LEVEL=6     # gzip level (1=fast, 9=smallest)
R2_ROOT=/opt/bucket     # filesystem root the script operates under
APP_DEPLOY_ROOT=/opt    # where /opt/<app>/.env markers live
```

### 4. Verify

```bash
cd /opt/bucket/storage/myordbok
r2.py info
```

Should show your R2 endpoint, bucket name, and detected DB container.

## How it knows what's an app

A folder under `/opt/bucket/storage/<n>/` is treated as an app **if and only if `/opt/<n>/.env` exists**. This `.env` file is created by the GitHub Actions deployment pipeline on every successful deploy.

This means:

- `myordbok/` → has `/opt/myordbok/.env` (deployed) → **is an app** ✓
- `access/` → no `/opt/access/.env` → **not an app** ✗
- `store/` → no `/opt/store/.env` → **not an app** ✗

No hardcoded list. Adding a new app requires no changes to the script.

If you try to run `r2.py backup mysql` from a non-app folder, the script refuses with a clear error pointing at the missing marker file.

## Layout convention

```
/opt/bucket/storage/
├── access/                    # shared secrets, not an app
│   ├── r2.conf
│   └── ...
├── myordbok/                  # an app (has /opt/myordbok/.env)
│   ├── mysql/
│   │   ├── latest.sql.gz      # auto-imported by docker-entrypoint-initdb.d
│   │   └── 2026-01-15T...sql.gz
│   ├── configs/
│   └── logs/
└── otherapp/                  # another app
    └── ...

/opt/myordbok/                 # app deployment dir
├── docker.production.yml
└── .env                       # deployment marker; created by GitHub Actions
```

## Commands

All commands run from inside an app folder by default. Use `--app <name>` to operate on a different app from anywhere.

### `backup mysql`

Dumps the live database from the running MySQL container, compresses it, and uploads to R2.

```bash
cd /opt/bucket/storage/myordbok
r2.py backup mysql
```

Produces two files locally and on R2:

- `mysql/latest.sql.gz` — always the most recent dump (overwritten each time)
- `mysql/<timestamp>.sql.gz` — historical dump (kept until pruned)

The script:

1. Finds the running container for stack `<app>_db`
2. Reads `DB_NAME` (or `MYSQL_DATABASE`) from the container's environment
3. Runs `mysqldump` inside the container — credentials never touch the host
4. Streams output through gzip to local files
5. Uploads both files to `s3://<bucket>/<app>/mysql/`

### `restore mysql`

Downloads `latest.sql.gz` from R2 to the local `mysql/` folder. Does not import — Docker's `docker-entrypoint-initdb.d` does that automatically when the container starts on an empty volume.

```bash
cd /opt/bucket/storage/myordbok
r2.py restore mysql

# Idempotent variant — only restore if local file is missing
r2.py restore mysql --if-empty
```

`--if-empty` is intended for chaining with deployment:

```bash
r2.py restore mysql --if-empty && docker stack deploy -c docker.production.yml myordbok
```

First run: downloads the dump. Subsequent runs: instant exit, no network call.

**Note:** restore needs zero MySQL credentials. It just downloads a file. This solves the chicken-and-egg problem of needing to populate `/docker-entrypoint-initdb.d` *before* the DB container starts.

### `list mysql`

Shows all dumps in R2 for this app, newest first.

```bash
r2.py list mysql
```

Output:

```
s3://storage/myordbok/mysql/
  2026-01-15 14:30:00 UTC    142.30 MB  2026-01-15T14-30-00Z.sql.gz
  2026-01-15 14:30:00 UTC    142.30 MB  latest.sql.gz
  2026-01-14 14:30:00 UTC    141.80 MB  2026-01-14T14-30-00Z.sql.gz
  ...
```

### `prune mysql`

Deletes old timestamped dumps from R2, keeping the N newest. `latest.sql.gz` is never touched.

```bash
r2.py prune mysql                # use RETENTION_KEEP from config (default 7)
r2.py prune mysql --keep 14      # keep last 14
r2.py prune mysql --keep 7 --dry-run    # preview without deleting
```

Run manually or via cron — backups never auto-prune.

### `push <folder>`

Uploads all files under `<folder>/` to R2. Recursive. Always overwrites.

```bash
cd /opt/bucket/storage/myordbok
r2.py push configs
r2.py push logs
r2.py push uploads --dry-run
```

R2 destination is `s3://<bucket>/<app>/<folder>/...`.

### `pull <folder>`

Downloads everything under the R2 prefix to local `<folder>/`. Recursive. Always overwrites local files.

```bash
cd /opt/bucket/storage/myordbok
r2.py pull configs
```

### `sync <source> <target>`

General-purpose copy between local and R2. Either side may be a local path or an `r2:bucket/key` URL — exactly one of each. No `--app`, no cwd inference; paths are explicit.

```bash
# Download
r2.py sync r2:storage/myordbok/configs/ /opt/foo/configs/

# Upload
r2.py sync /opt/foo/configs/ r2:storage/myordbok/configs/

# Single file with rename
r2.py sync r2:storage/dumps/big.tar.gz /tmp/renamed.tar.gz

# Different destination name
r2.py sync r2:storage/myordbok/configs /opt/anywhere/whatever
```

Trailing slashes are decoration and ignored. `s3:` is accepted as a synonym for `r2:`.

The destination is whatever you wrote — folder source contents land directly inside the destination, no extra folder level inserted.

### `status`

Lists subfolders in the current app folder and how each will be handled.

```bash
cd /opt/bucket/storage/myordbok
r2.py status
```

### `info`

Diagnostic output: detected app, R2 endpoint, R2 bucket connectivity, MySQL container, etc. Useful for first-time setup verification and debugging.

```bash
r2.py info
```

## Flags

### Global flags

| Flag         | Effect                                                                   |
| ------------ | ------------------------------------------------------------------------ |
| `--config`   | Path to config file (default: `/opt/bucket/storage/access/r2.conf`)      |
| `--dry-run`  | Show what would happen without making changes                            |
| `-h` / `--help` | Show help                                                             |

### Per-command flags

| Flag         | Available on              | Effect                                                       |
| ------------ | ------------------------- | ------------------------------------------------------------ |
| `--app NAME` | all (except `sync`)       | Operate on this app instead of detecting from cwd            |
| `--if-empty` | `restore`                 | Only restore if local target is missing (idempotent)         |
| `--keep N`   | `prune`                   | How many timestamped dumps to keep                           |

`--app` example:

```bash
# These are equivalent
cd /opt/bucket/storage/myordbok && r2.py push configs
r2.py push --app myordbok configs       # works from any directory
```

## Typical workflows

### Routine backup (cron)

```cron
# Daily at 3 AM
0 3 * * * cd /opt/bucket/storage/myordbok && /usr/local/bin/r2.py backup mysql

# Weekly retention
0 4 * * 0 cd /opt/bucket/storage/myordbok && /usr/local/bin/r2.py prune mysql --keep 7
```

### First-time deploy on a new server

```bash
# Restore the most recent backup, then start the stack
cd /opt/bucket/storage/myordbok
r2.py restore mysql --if-empty
cd /opt/myordbok
docker stack deploy -c docker.production.yml --with-registry-auth myordbok
```

If no backup exists yet (first deploy ever), `--if-empty` exits cleanly and the stack starts with an empty database.

### Migrating to a new server

```bash
# On new server, after rsync'ing /opt/bucket/storage/ from old server
sudo apt install python3-boto3
sudo wget https://raw.githubusercontent.com/khensolomon/lets/make/server/r2.py -O /usr/local/bin/r2.py
sudo chmod +x /usr/local/bin/r2.py
sudo cp /opt/bucket/storage/access/r2.conf.example /opt/bucket/storage/access/r2.conf
sudo nano /opt/bucket/storage/access/r2.conf      # paste credentials

# Verify
cd /opt/bucket/storage/myordbok
r2.py info

# Pull latest backup if needed
r2.py restore mysql
```

### Backing up arbitrary config files

```bash
cd /opt/bucket/storage/myordbok
r2.py push configs
```

### Copying files outside the app convention

For one-off or ad-hoc copies that don't fit the per-app layout:

```bash
# Back up a Cloudflare credentials file
r2.py sync /opt/bucket/storage/access/r2.conf r2:storage/backups/r2.conf

# Pull a shared resource down
r2.py sync r2:storage/shared/nltk-data.tar.gz /opt/nltk-data.tar.gz
```

## Conventions and assumptions

- **App = `/opt/<n>/.env` exists.** Folders without this marker are not apps.
- **Stack name = app name.** Deploy with `docker stack deploy ... <appname>` matching the folder name.
- **DB service = `<app>_db`.** Auto-detected via Swarm labels, with name-prefix fallback for plain Compose.
- **DB credentials live in the container, not on the host.** `MYSQL_ALLOW_EMPTY_PASSWORD=yes` works; `MYSQL_ROOT_PASSWORD` and `DB_ROOT_PWD` env vars are also tried as fallbacks.
- **R2 key layout = `<bucket>/<app>/<folder>/...`** — bucket name from `r2.conf`, app and folder from cwd.
- **Compression = gzip.** Native to MySQL's `docker-entrypoint-initdb.d` for auto-import on init.

## Troubleshooting

**`error: '<app>' is not a deployed app on this host.`**
The script can't find `/opt/<app>/.env`. Either you're not in an app folder, or the deployment hasn't happened yet on this server. For ad-hoc operations, use `r2.py sync` with explicit paths instead.

**`error: No running container found for service '<app>_db'.`**
The stack isn't running. Deploy it first with `docker stack deploy ...`, or check `docker service ls` to see what's actually up.

**`error: Could not read DB_NAME/MYSQL_DATABASE from container env.`**
The container is running but doesn't have these env vars. Check `docker exec <container> printenv | grep -E 'DB_NAME|MYSQL_DATABASE'`. Most likely your compose file isn't passing them through.

**`error: All mysqldump strategies failed.`**
The script tried root with no password, then `MYSQL_ROOT_PASSWORD`, then `DB_ROOT_PWD`. None worked. Check the container's env vars with `docker exec <container> printenv | grep -i pass`.

**`error: cwd ... is not under R2_ROOT ...`**
You're running from outside the configured `R2_ROOT` (default `/opt/bucket`). Either `cd` into an app folder under R2_ROOT, or use `--app <name>` to specify which app to operate on.

**`error: boto3 is required but not installed.`**
Run the install command for your system (see Install section above).

## Security notes

- `r2.conf` should be `chmod 600` and root-owned. The script warns if it isn't.
- The script never reads MySQL credentials from the host. It uses `docker exec` to run `mysqldump` inside the container, where credentials already live in the container's environment.
- R2 access keys can be scoped to specific buckets in the Cloudflare dashboard. Recommended for production tokens.
- `latest.sql.gz` and timestamped dumps in R2 are encrypted at rest by R2. In-flight transfers use HTTPS.
